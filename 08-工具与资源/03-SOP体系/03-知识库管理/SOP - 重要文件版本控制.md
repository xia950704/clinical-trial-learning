# SOP - 重要文件版本控制

> **目的**：建立合同、方案等重要文件的版本控制规范，避免重复文件、版本混乱、链接损坏

---

## 一、适用范围

| 文件类型 | 说明 | 版本控制级别 |
|---------|------|------------|
| **临床试验方案** | V1.0/V2.0/V3.0等 | 🔴 最高（必须严格） |
| **合同文件** | 主合同/补充协议 | 🔴 最高（必须严格） |
| **伦理批件** | 初始批件/年审批件 | 🟡 中等（按时间线） |
| **eCRF/CRF** | 填写指南/结构定义 | 🟡 中等（按版本） |
| **SAP/SAR** | 统计分析计划/报告 | 🟡 中等（按版本） |
| **知情同意书** | ICF版本 | 🟡 中等（按版本） |
| **项目索引/总览** | 项目导航文件 | 🟢 常规（定期更新） |

## 三、文件命名规范

### 3.1 命名格式

```
[项目编号]-[文件类型]-[版本/日期]-[描述].md
```

| 元素 | 格式 | 示例 |
|------|------|------|
| 项目编号 | 数字+短横线 | `274-` |
| 文件类型 | 中文简写 | `合同`、`方案`、`伦理` |
| 版本 | V+数字+日期 | `V3.0`、`V2.0-20210909` |
| 描述 | 中心名/内容简述 | `安贞主合同`、`各中心汇总` |

### 3.2 命名示例

| ✅ 正确 | ❌ 错误 |
|--------|--------|
| `274-合同-安贞主合同V20211217.md` | `274 安贞合同 2021.12.17.md` |
| `274-方案-V3.0-核心摘要.md` | `274 临床试验方案 V3.0.md` |
| `274-伦理-各中心批件汇总.md` | `274 各中心伦理批件汇总.md` |
| `274-合同汇总.md` | `274合同汇总_001.md` |

### 3.3 特殊规则

1. **无空格**：文件名中不使用空格，用短横线 `-` 连接
2. **无特殊字符**：不使用 `>>>`、`<<<`、`_` 等
3. **版本优先**：同一文件多个版本，按版本号命名
4. **汇总优先**：汇总类文件放在 `01-索引/` 或 `09-报告/`

## 四、版本控制流程

### 4.1 新文件创建

```
1. 搜索现有文件（避免重复）
   └─ 搜索关键词：项目编号+文件类型+中心名
   
2. 确认无重复后创建新文件
   └─ 使用标准命名格式
   └─ 添加标准frontmatter
   
3. 更新项目索引
   └─ 在274项目核心文件索引.md中添加链接
   
4. 建立交叉链接
   └─ 关联相关方案/合同/伦理文件
```

### 4.2 文件更新

```
1. 确认文件版本
   └─ 检查frontmatter中的version字段
   
2. 更新内容
   └─ 保留版本历史（在文件头部添加变更记录）
   
3. 更新frontmatter
   └─ updated: 2026-06-03
   └─ version: V3.0
   
4. 通知相关方（如适用）
   └─ 在AI观察日志中记录变更
```

### 4.3 版本变更记录

在重要文件头部添加变更记录表：

```markdown
## 版本历史

| 版本 | 日期 | 变更内容 | 变更人 |
|------|------|---------|--------|
| V1.0 | 2021-09-09 | 初始版本 | - |
| V2.0 | 2021-09-09 | 方案修订 | - |
| V3.0 | 2022-03-14 | 最新主版本 | - |
```

## 五、frontmatter标准

### 5.1 必填字段

```yaml
---
title: [文件标题]
tags: [标签1, 标签2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: active|archived|draft
priority: P0|P1|P2
---
```

### 5.2 项目文件专用字段

```yaml
---
title: 274-合同-安贞主合同
tags: [274项目, 合同, 尼非卡兰]
created: 2026-05-30
updated: 2026-06-03
status: active
priority: P0
project: 274-核心RCT-尼非卡兰vs胺碘酮
version: V20211217
contract_number: HT2112357
center: 北京安贞医院
source: 移动硬盘/项目资料/274/274合同/安贞01/
---
```

### 5.3 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | ✅ | 文件标题 |
| `tags` | ✅ | 标签（至少包含项目编号） |
| `created` | ✅ | 创建日期 |
| `updated` | ✅ | 最后更新日期 |
| `status` | ✅ | 状态：active/archived/draft |
| `priority` | ✅ | 优先级：P0/P1/P2 |
| `project` | ✅ | 所属项目 |
| `version` | ✅ | 版本号 |
| `source` | ✅ | 原始文件位置 |

## 六、重复文件处理

### 6.1 识别重复

```bash
# 搜索相同标题的文件
搜索: "274合同汇总"
搜索: "274 项目综合资料处理报告"
```

### 6.2 处理流程

```
1. 识别所有重复版本
   └─ 记录每个文件的路径、内容、链接状态
   
2. 确定权威版本
   └─ 内容最完整
   └─ 链接状态正常
   └─ frontmatter完整
   
3. 删除重复版本
   └─ 使用mcp_obsidian_brain_delete_note
   └─ 确认: true
   
4. 更新链接
   └─ 将所有指向已删除文件的链接更新为指向权威版本
```

| 2026-06-03 | 创建SOP | 初始版本 |
| 2026-06-03 | 合并重复文件 | 删除6个重复版本 |
| 2026-06-03 | 更新链接 | 更新7处链接 |
| 2026-06-03 | 统一目录结构 | 移动20个274项目文件至09-百利/12-274项目/目录 |

```
1. 检查incoming链接
   └─ 确认哪些文件引用了该文件
   
2. 检查outgoing链接
   └─ 确认该文件引用了哪些文件
   
3. 确认无孤立文件
   └─ 删除后不会导致其他文件链接损坏
```
