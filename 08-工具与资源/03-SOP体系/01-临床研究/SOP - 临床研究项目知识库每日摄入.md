# SOP - 临床研究项目知识库每日摄入（V1.0）

> 版本：V1.0 | 编制日期：2026-06-02
> 适用场景：临床研究项目（如274/2000/240/756/IIT）的PDF文件批量提取并生成Obsidian笔记
> 来源：274项目知识库综合摄入报告 + 已验证流程模式 + AI协作踩坑录
> 关联：[[05-临床研究/IIT 数据分析与学术文章撰写 SOP（V2.0）.md]] [[02-百利/06-进度追踪/PDF处理工具链.md]] [[04-方法学与工具/00-踩坑录.md]]

## 一、目标与交付物

**目标**：每日自动扫描项目目录新增/变更PDF文件，提取文本并生成结构化Obsidian笔记，保持知识库与原始文件同步。

**交付物清单**：
- 结构化Obsidian笔记（合同/病例/安全性/会议/文献/伦理等分类）
- 每日摄入报告（PDF工具状态、处理统计、差异分析）
- manifest进度追踪文件（各项目的processed_files计数）
- 跳过文件清单（扫描件/乱码/空文件）

**质量标准**：
- 笔记必须写入vault，processed/仅为中转
- manifest计数≠vault实际笔记数时，必须标注差异及原因
- 扫描件必须标记 `[SKIP_SCANNED]`
- 每篇笔记frontmatter包含：title/tags/status/project_id/source_file

---

## 二、标准工序（6个阶段）

### 阶段1：预检与环境准备

**目标**：确认工具链和环境就绪

**动作顺序**：
1. 验证pdftotext版本（`pdftotext -v`，需≥v22.x）
2. 验证PyMuPDF可用性（sandbox环境可能不可用，属预期）
3. 验证硬盘挂载（`/Volumes/夏爽/_当前项目/项目资料` 可访问）
4. 读取各项目 manifest.yaml / progress-note.md
5. 扫描目录，筛选出新增/变更的PDF文件

**输入**：项目根目录、manifest文件
**输出**：待处理文件列表（含优先级排序）

**踩过的坑**：
- ⚠️ 硬盘未挂载导致路径不可访问。→ 预检阶段先检查挂载状态，未挂载时跳过并记录。
- ⚠️ manifest计数≠vault实际笔记数。→ 274项目manifest报告processed_files: 91，vault实际合同笔记仅72篇，差异19篇（manifest计数的是LLM尝试读取数，非实际写入数）。→ 必须分别追踪manifest计数和vault实际写入数。

---

### 阶段2：文件扫描与优先级排序

**目标**：按优先级确定处理顺序

**动作顺序**：
1. 扫描项目目录所有PDF文件
2. 按优先级分类：
   - **P0**：合同/SAE（最高优先级，影响合规）
   - **P1**：protocol/伦理资料（核心文件）
   - **P2**：费用/会议记录
   - **P3**：文献/病例（批量处理）
3. 按P0→P1→P2→P3排序
4. 标记新增文件 vs 变更文件

**输入**：待处理文件列表
**输出**：排序后的处理队列

**踩过的坑**：
- ⚠️ 扫描件占比过高（274项目32.7%为扫描件）。→ 预评估阶段输出direct通道 vs ocr_needed通道比例，超过30%时触发OCR积压告警。

---

### 阶段3：PDF文本提取

**目标**：从PDF提取可编辑文本

**动作顺序**：
1. 对每个PDF执行 `pdftotext -layout input.pdf -` 提取到stdout
2. 检查提取字符数：
   - ≥50字符 → 进入阶段4（LLM提取）
   - <50字符 → 标记 `[SKIP_SCANNED]`，输出到跳过清单
3. 若pdftotext输出乱码或空白 → 回退到PyMuPDF（如可用）
4. 若PyMuPDF也输出<50字符 → 标记为OCR需要，加入OCR积压清单

**输入**：排序后的处理队列
**输出**：提取文本 + 跳过清单

**踩过的坑**：
- ⚠️ 纯扫描件PDF（全页为图片）两个工具都无法提取。→ 累计OCR积压145+文件，需部署OCRmyPDF批量处理。
- ⚠️ 部分PDF提取字符极少（如`01013 邓玉兰.pdf`仅28字符）。→ 低于阈值直接跳过，不浪费LLM调用。

---

### 阶段4：LLM结构化提取

**目标**：从提取文本中按模板提取关键信息

**动作顺序**：
1. 根据文件类型选择对应模板：
   - 合同模板：提取甲方/乙方/金额/日期/条款要点
   - 病例模板：提取SUBJID/中心/诊断/用药/AE/转归
   - SAE模板：提取事件描述/严重程度/因果关系/处理措施
   - 会议模板：提取时间/参会人/决策项/待办
   - 文献模板：提取标题/作者/期刊/结论/关键数据
2. 区分【原文】【数据】【推断】三类内容
3. 写入frontmatter（project_id、source_file、read_status）

**输入**：提取文本 + 文件类型
**输出**：结构化笔记内容

**踩过的坑**：
- ⚠️ LLM尝试读取但写入失败（manifest计数91 vs vault实际72）。→ 区分"尝试读取"和"实际写入"，manifest只追踪前者，vault追踪后者。

---

### 阶段5：写入Vault与建图

**目标**：将笔记写入vault并建立关联链接

**动作顺序**：
1. 按项目目录结构写入笔记（如`09-百利/01-文献/`、`09-百利/02-合同/`）
2. 建立wikilink：
   - 合同笔记 → 关联项目主笔记
   - 病例笔记 → 关联中心笔记 + 受试者笔记
   - 文献笔记 → 关联主题索引
3. 更新manifest进度追踪
4. 验证写入（read_file确认内容完整）

**输入**：结构化笔记内容
**输出**：vault中的笔记文件 + 更新后的manifest

**踩过的坑**：
- ⚠️ 病例目录未纳入manifest追踪。→ 274项目manifest只追踪合同，病例60篇完全独立追踪。→ manifest需扩展为多目录追踪。

---

### 阶段6：报告输出

**目标**：输出当日摄入报告

**报告内容**：
1. PDF工具状态（文字描述，禁止表格）
2. 硬盘挂载状态
3. 可读性预评估（direct vs ocr_needed比例）
4. 各项目处理详情（新增/跳过/累计）
5. manifest计数 vs vault实际笔记数差异
6. 信息缺口清单（高/中/低优先级）
7. 今日产出汇总（新增笔记数 + 分类明细）

**输入**：各阶段输出数据
**输出**：每日摄入报告笔记

## 三、质量控制检查点

| QC编号 | 检查点 | 阶段 | 阈值 |
|:------:|:------|:----:|:----:|
| QC-1 | 硬盘挂载状态 | 阶段1 | 必须可访问 |
| QC-2 | pdftotext可用性 | 阶段1 | 版本≥v22.x |
| QC-3 | 提取字符数阈值 | 阶段3 | ≥50字符才进入LLM |
| QC-4 | manifest vs vault差异 | 阶段5 | 差异必须标注原因 |
| QC-5 | 扫描件占比告警 | 阶段2 | >30%触发OCR积压告警 |
| QC-6 | 笔记frontmatter完整性 | 阶段5 | 必须含project_id/source_file |

## 四、紧急情况处理机制

| 场景 | 处理流程 | 回退级别 |
|:----|:---------|:--------:|
| A 硬盘未挂载 | ①检查挂载 → ②手动挂载 → ③重新预检 → ④若仍失败则跳过当日摄入 | 当日任务暂停 |
| B OCR积压超过200文件 | ①输出积压清单 → ②评估批量OCR可行性 → ③部署ocrmypdf → ④分批次处理 | 阶段3回退至OCR队列 |
| C manifest与vault差异过大 | ①逐文件比对 → ②找出未写入原因 → ③补写或标记为跳过 → ④更新manifest | 阶段5回退 |
| D pdftotext版本过低 | ①检查版本 → ②brew upgrade poppler → ③验证 → ④重新提取 | 阶段1回退 |

## 六、OCR积压处理（待改进项）

> 当前累计OCR积压：145+文件（含历史）

**待部署方案**：
1. 安装 tesseract-ocr：`brew install tesseract`
2. 安装 ocrmypdf：`pip3 install ocrmypdf`（确保ghostscript已安装：`brew install ghostscript`）
3. 批量处理脚本：
   ```bash
   for f in *.pdf; do
     ocrmypdf --lang chi_sim+eng "$f" "ocr_${f}"
   done
   ```
4. 处理后的OCR PDF重新进入阶段3流程
