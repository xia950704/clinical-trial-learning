## 三、子流程1：SOP自我审查与迭代

### 3.1 目标

审查已有SOP的有效性，识别僵尸SOP、过时SOP、可合并SOP。

### 3.2 动作顺序

**Step 1：扫描SOP清单（≤5次）**

```
1. 列出所有SOP文件：mcp_obsidian_brain_list_notes(directory="04-方法学与工具/90-SOP体系")
2. 列出所有流程模式：mcp_obsidian_brain_list_notes(directory="04-方法学与工具/90-流程方法论/03-")
3. 读取每个SOP的frontmatter（version、created、updated）
4. 输出SOP清单：文件名 | version | 创建日期 | 最后更新 | 状态
```

**Step 2：分析使用频率（≤10次）**

```
1. 对每个SOP，用session_search搜索最近30天是否被引用
   - 搜索关键词：SOP文件名 + "SOP" + 核心关键词
2. 统计：引用次数 / 最近引用日期 / 是否被跳过
3. 标记：
   - 高频（≥5次/月）→ 保留
   - 低频（1-4次/月）→ 观察
   - 零引用（0次/30天）→ 僵尸SOP
```

**Step 3：审查内容时效性（≤10次）**

```
1. 对每个SOP，检查：
   - frontmatter updated日期是否>90天
   - 是否有过时的工具/命令/URL
   - 是否有已被替代的流程
2. 标记：
   - 时效OK（<90天）→ 保留
   - 需更新（90-180天）→ 待审查
   - 过时（>180天）→ 建议更新
```

**Step 4：识别合并机会（≤5次）**

```
1. 分析SOP之间的重叠：
   - 两个SOP是否处理相似场景？
   - 是否有SOP可以被另一个SOP覆盖？
2. 输出合并建议：SOP A + SOP B → 合并为SOP C
```

**Step 5：输出审查报告（≤1次）**

```
输出格式：
【SOP审查报告】
总SOP数：X
高频（保留）：Y个
低频（观察）：Z个
零引用（僵尸）：W个 → 建议删除/归档
需更新：V个
可合并：U对

待确认操作：
- [ ] 删除/归档：SOP X, SOP Y
- [ ] 更新：SOP A (原因: ...)
- [ ] 合并：SOP B + SOP C → SOP D
```

## 四、子流程2：知识缺口主动填补

### 4.1 目标

自动发现vault中的知识缺口（broken links、stub、缺失frontmatter、内容过短），并填补。

### 4.2 动作顺序

**Step 1：扫描缺口（≤10次）**

```
1. 扫描broken links：
   - mcp_obsidian_brain_list_notes(includeStubs=true)
   - 识别frontmatter _stub: true 的笔记
2. 扫描缺失frontmatter：
   - search_files(pattern="*.md", target="content", path="01-股票投研")
   - 检查每篇笔记是否含 "---" frontmatter
3. 扫描内容过短：
   - 对每篇笔记，检查字符数 < 200
4. 输出缺口清单：文件 | 缺口类型 | 严重程度(P0/P1/P2)
```

**Step 2：评估填补优先级（≤5次）**

```
1. 对每个缺口，评估：
   - P0：broken link指向重要主题（如核心股票、核心SOP）
   - P1：stub笔记有明确主题但内容缺失
   - P2：frontmatter缺失或内容过短
2. 排序：P0 → P1 → P2
```

**Step 3：生成填补内容（≤20次）**

```
对每个P0/P1缺口：
1. 读取缺口笔记的上下文（linked notes）
2. 根据上下文生成填补内容：
   - stub → 创建笔记草稿（用对应模板）
   - broken link → 搜索vault或web查找目标，如找不到则标记"需人工确认"
   - frontmatter缺失 → 自动补全frontmatter
   - 内容过短 → 根据主题自动补充（如股票笔记补充关键数据表）
3. 生成预览（dryRun=true），不直接写入
```

**Step 4：用户确认（≤5次）**

```
1. 输出填补预览清单
2. 用户确认/修改/拒绝每个填补
3. 对确认的填补，执行写入
```

**Step 5：验证写入（≤5次）**

```
1. 对每个写入的笔记，read_file验证内容完整性
2. 检查wikilink是否正确建立
3. 输出验证报告
```

## 七、速率控制层（核心机制）

### 7.1 实现方式

**方案A：Python脚本控制（推荐）**

```python
import time
from hermes_tools import terminal

class RateLimiter:
    def __init__(self, max_rpm=3):
        self.max_rpm = max_rpm
        self.min_interval = 60 / max_rpm  # 最小间隔（秒）
        self.last_call = 0
        self.call_count = 0
        self.window_start = time.time()
    
    def wait_if_needed(self):
        now = time.time()
        # 检查5小时窗口是否重置
        if now - self.window_start > 18000:  # 5小时
            self.call_count = 0
            self.window_start = now
        
        # 检查分钟窗口
        if self.call_count >= self.max_rpm:
            elapsed = now - self.window_start
            wait_time = 300 - elapsed  # 等待到5小时窗口重置
            if wait_time > 0:
                print(f"⚠️ 速率限制：等待{wait_time:.0f}秒")
                time.sleep(wait_time)
            self.call_count = 0
            self.window_start = time.time()
        
        # 检查最小间隔
        time_since_last = now - self.last_call
        if time_since_last < self.min_interval:
            wait = self.min_interval - time_since_last
            time.sleep(wait)
        
        self.last_call = time.time()
        self.call_count += 1
        print(f"📊 调用计数: {self.call_count}/{self.max_rpm} (RPM)")

# 使用示例
limiter = RateLimiter(max_rpm=3)

def safe_tool_call(tool_func, *args, **kwargs):
    limiter.wait_if_needed()
    return tool_func(*args, **kwargs)
```

### 7.2 429错误处理

```python
def handle_429(error):
    """处理429速率限制错误"""
    if "429" in str(error) or "rate limit" in str(error).lower():
        # 解析Retry-After头部（如有）
        import re
        match = re.search(r'Retry-After: (\d+)', str(error))
        if match:
            wait_seconds = int(match.group(1))
        else:
            wait_seconds = 300  # 默认等待5分钟
        
        print(f"⚠️ 429速率限制，等待{wait_seconds}秒后重试")
        time.sleep(wait_seconds)
        return True  # 需要重试
    return False  # 不需要重试
```
